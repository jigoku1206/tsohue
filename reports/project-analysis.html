重大問題

Fresh database setup 可能直接失敗

Critical

證據：supabase/schema.sql:30-33 先建立 recurring_rules，其中 ledger_id uuid references public.ledgers(id)；但 ledgers 表到 supabase/schema.sql:112 才建立。

風險：PostgreSQL 無法建立指向尚不存在 table 的 foreign key。README 指示新專案直接執行整份 schema，fresh install 可能卡住，與檔案註解「safe to run multiple times」不一致。

建議：把 ledgers / ledger_members 提前建立，或先建立 recurring_rules.ledger_id 欄位但延後到 ledgers 存在後再 alter table add constraint。

重跑 schema 會清掉所有週期交易關聯

Critical

證據：supabase/schema.sql:55 執行 update public.transactions set recurring_id = null where recurring_id is not null;，不是只清孤兒資料。

風險：任何維運或升級時重跑 schema，都會讓所有交易失去 recurring rule 關聯，後續「編輯全部週期 / 此筆及之後」會失準，甚至重新產生重複資料。

建議：改成只清掉 orphaned records，例如檢查 not exists (select 1 from recurring_rules where id = transactions.recurring_id)；更好是改用版本化 migration，不把資料修補與建表混在同一份 bootstrap SQL。

交易 RLS 讀取規則過寬，可能造成資料外洩

Critical

證據：supabase/schema.sql:213-222 的 transactions_select 允許所有 ledger_id is null 的交易被讀取；公用帳本分支也沒有明確限制 to authenticated 或 auth.role() = 'authenticated'。

風險：舊資料或個人交易若是 ledger_id = null，任何通過 API policy 的使用者都可能讀到。若 Supabase anon role 可查表，公用帳本交易也可能在未登入狀態被 anon key 讀取。

建議：將 null-ledger 條件改成 ledger_id is null and user_id = auth.uid()；公用帳本若只給登入者看，policy 應指定 to authenticated 或加入 auth.role() = 'authenticated'。建議抽出 can_access_ledger(ledger_id) helper，讓 select/insert/update/delete 的權限語意一致。

註冊開關只在 UI 檢查，Server Action 可被繞過

High

證據：app/register/page.tsx 會呼叫 getRegistrationEnabled() 決定是否顯示表單；但 app/actions/auth.ts:26-63 的 register 沒有檢查 registration_enabled。

風險：Server Actions 是伺服器端入口，不能只靠頁面是否顯示表單保護。當管理員關閉註冊時，仍應在 action 層拒絕請求。

建議：在 register 一開始讀取 app setting，不開放時直接回傳錯誤；同時在測試中覆蓋「關閉註冊仍直接呼叫 action」的情境。

正式依賴有高風險安全公告

High

證據：npm audit --omit=dev 回報 11 vulnerabilities，其中 4 high。包含 next@16.2.1 的多個 advisory，audit 建議升到 next@16.2.6，另有 fast-uri、path-to-regexp、picomatch 等 high 風險相依套件。

風險：Next.js 屬於應用核心 runtime；middleware/proxy bypass、cache poisoning、DoS、SSRF、XSS 類公告需要優先評估，尤其此專案依賴 App Router、Proxy、Server Components。

建議：先建立回歸測試，再升級 Next 與相依套件；若升版牽動 React/Next 行為，至少覆蓋登入保護、dashboard 資料讀寫、CSV 匯入、週期交易、admin setting。

潛在問題

個資曝光面偏大：所有登入者可讀全部 profile email

High

證據：supabase/schema.sql:101-103 允許所有 authenticated user select profiles；app/actions/ledgers.ts:99-117 的 getAllUsers() 會列出其他使用者的 nickname 與 email。

風險：若使用者數增加，email 清單會成為不必要的個資暴露。共同帳本邀請不一定需要全量使用者列表。

建議：改成 owner 輸入 email 後做 exact lookup，或建立 profiles_public view 只暴露必要欄位；email 顯示可限制給 admin 或已有共同帳本關係者。

Server Actions 缺少一致的輸入驗證

High

證據：app/actions/transactions.ts:82-103、app/actions/import.ts:41-54、app/actions/recurring.ts:96-133 直接使用 FormData 或 client-side 解析結果，未集中驗證 date、currency、amount、category、note 長度、recurring frequency 等。

風險：前端已做部分 UX 檢查，但 action 仍可能收到非 UI 產生的請求。資料庫會擋下一些錯誤，但使用者會看到 raw DB error，也可能留下語意不合法但 DB 接受的資料。

建議：用 Zod、Valibot 或自訂 parser 建立 action schema；資料庫也補上 currency in (...)、char_length、exchange_rate > 0、合理 date range 等 constraint。

CSV 匯入權限與共同帳本成員模型不一致

Medium

證據：app/actions/import.ts:31-38 只允許 owner 或 public ledger 匯入；但 supabase/schema.sql:224-236 的交易 insert policy 允許 ledger member 寫入。

風險：共享帳本成員可以新增單筆交易，卻不能 CSV 匯入同一帳本，會造成產品行為不一致。

建議：權限檢查改用同一個 can_access_ledger 或查 ledger_members；也可直接依賴 RLS insert 結果，但錯誤訊息需要轉成友善文字。

週期交易修改/刪除不是 transaction，且多處忽略 Supabase error

Medium

證據：app/actions/recurring.ts:175-213 會先更新舊 rule、再 insert 新 rule、再更新 transactions；多個 await 沒有檢查 error。deleteRecurringByScope 也有相同模式。

風險：中間任一步失敗會造成 rule 與交易資料不一致；使用者仍會收到 { error: null }。

建議：將複合操作移到 PostgreSQL RPC function，以單一 DB transaction 執行；或至少每一步檢查 error 並回滾/補償。

共同帳本付款人用文字欄位，長期會難以追蹤

Medium

證據：transactions.paid_by 是 text；新增與編輯交易都存 display name。

風險：使用者改暱稱、重名、或手動輸入不同字串後，付款人統計會失準。成員列表目前也不包含 owner 本身，付款人選單可能與實際帳本成員不一致。

建議：新增 paid_by_user_id FK，UI 顯示時 join profile nickname；既有文字欄位可保留為 legacy display cache。

註冊流程對 Supabase Email Confirmation 不穩

Medium

證據：app/actions/auth.ts:35-47 在 signUp 後立即 insert profiles。若 Supabase 專案開啟 email confirmation，signUp 可能不建立可用 session，RLS 的 auth.uid() = id 會使 profile insert 失敗。

風險：不同 Supabase 設定下註冊結果不一致；使用者可能已建立 auth user，但 profile 沒建立，造成登入後 dashboard profile 缺失。

建議：用 DB trigger 在 auth.users 建立 profile，或在 callback/首次登入補建 profile；文件也應明確說明 email confirmation 設定。

公用帳本建立存在競態條件

Medium

證據：app/actions/auth.ts:49-59 與 app/dashboard/transaction-section.tsx 都用「先 count 再 insert」建立公用帳本。

風險：兩個請求同時看到 count=0 時，第二個 insert 會撞上 one_public_ledger_idx。目前其中一處甚至忽略 insert error。

建議：把公用帳本 seed 放到 migration，或使用單一 DB function / insert ... on conflict do nothing 類型的 idempotent 操作。

React 19 lint 規則已抓到多個效能與正確性風險

Medium

證據：npm run lint 失敗。重點包含 app/dashboard/demo-dashboard.tsx 與 components/amount-calculator.tsx 在 render 階段寫 ref；components/add-transaction-dialog.tsx、components/edit-transaction-dialog.tsx、components/calendar-view.tsx 在 effect 中同步 setState。

風險：目前 build 可過，但 lint 已反映 React Compiler / React 19 推薦模型下的不穩定寫法，之後升級或啟用更嚴格最佳化時容易踩雷。

建議：把 render-time ref assignment 移到 effect 或 useEvent-like helper；同步 derive state 的情境優先改成 render-time derived value 或 controlled state reset key。

可優化方向

建立 migration 與 RLS 測試防線

High ROI

目前 schema 是單檔 bootstrap，但已混入建表、資料修補、policy、seed。建議導入 Supabase CLI migration，並加入 local Supabase 或 pgTAP 測試：

  • fresh database 從 0 跑 migration 應通過。
  • 重跑 migration 不應修改既有 transaction/recurring relation。
  • user A 不可讀 user B 的 null-ledger transactions。
  • anon 不可讀需要登入的 public ledger transactions。
  • ledger member 可新增、匯入、讀取共享帳本交易；非 member 不可。

收斂權限邏輯，避免 app 與 DB 各說各話

High ROI

目前權限散在 RLS policy、Server Actions 手寫查詢、UI 顯示條件。建議以資料庫 helper 為單一真相，例如 can_read_ledger、can_write_ledger、is_ledger_owner，Server Action 只做 user intent 與友善錯誤包裝。

補齊 CI：lint、typecheck、build、audit、最小 E2E

Medium ROI

目前缺少 CI 與測試。建議新增 GitHub Actions 或等價流程：npm ci、npm run lint、npm run build、npm audit --omit=dev --audit-level=high。E2E 先覆蓋登入保護、建立交易、切換月份、CSV 匯入、週期交易。

改善 build 穩定性與部署可預期性

Medium ROI

Next.js build 顯示 workspace root 推斷到 /Users/jerry/package-lock.json，建議在 next.config.ts 設定 Turbopack root 或移除上層 lockfile 影響。Google Fonts 也讓離線或受限網路 build 失敗，若部署環境可能受限，可改用 self-hosted font 或本地 fallback。

資料模型正規化

Medium ROI

可逐步把 paid_by、category、subcategory 從純文字轉成 FK 或 snapshot + FK 混合模型。短期可先保留文字欄位，新增 paid_by_user_id、category_id，用 migration 回填再讓 UI 使用新欄位。

效能與 UX 細節

Nice to have
  • getTransactions 與 range export 目前 select('*')，可改列出必要欄位，並為大量交易加入分頁或 streaming export。
  • ensureRecurringForMonth 每次讀月份都可能觸發，可改成 DB RPC + on conflict do nothing，避免併發 insert error 被吞掉。
  • getCategories 建樹時用多次 filter，資料量大時可先 group by parent_id。
  • 匯率快取目前是 process memory；正式多 instance 部署時快取命中不穩，可接受，但若 API 限流可考慮 Edge/DB cache。

建議處理順序

優先級 工作 完成定義
P0 修正 schema 外鍵順序、移除 destructive recurring_id update、收緊 transactions RLS。 fresh migration 通過；重跑不改資料；RLS 測試證明 user/anon 無法越權讀交易。
P0 在 register action 重新檢查 registration setting。 關閉註冊時，即使直接呼叫 action 也無法建立帳號。
P1 升級 Next 與 vulnerable dependencies，修 lint errors。 npm audit --omit=dev --audit-level=high 無 high；npm run lint 通過。
P1 建立 Server Action validation 與友善錯誤。 交易、週期交易、CSV 匯入、帳本設定都有 schema validation；不回傳 raw DB error 給使用者。
P2 補測試與 CI。 PR 必跑 lint/build/RLS tests；核心流程有 E2E smoke test。
P2 改善 profile privacy、paid_by 正規化、public ledger seed。 email 不再全量暴露；付款人可追蹤 user id；公用帳本由 migration 或 idempotent RPC 建立。
建議第一個修補 PR：先處理 supabase/schema.sql 的三個 P0 問題，因為它們同時影響新部署、既有資料安全與後續維運可靠性。這部分修好後，再進入 action validation 與依賴升級會比較穩。
